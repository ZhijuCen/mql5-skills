# MQL5 Economic Calendar — Binary File Format Quick Reference

**Source**: Article 22196 — "MetaTrader 5 and the MQL5 Economic Calendar:
How to Turn News into a Reproducible Trading System"
<https://www.mql5.com/en/articles/22196>

**Why this exists**: The MetaTrader 5 strategy tester does NOT have
internet access. Calling `CalendarValueHistory()` from an EA inside the
tester returns an empty array and an error. The pattern from article
22196 is to **pre-export the calendar to a binary file once** (live
terminal, where `CalendarValueHistory()` works), then **embed it as a
`#resource` array of `MqlCalendarValue`** in the EA, so the tester
reads from RAM instead of trying to reach the server.

This document describes the on-disk layout of that binary file so you
can inspect / re-serialize it from outside the MetaTrader environment
(e.g. from a Python script that needs to align EA backtests with news
timestamps without internet access).

The repository's parser lives at
`scripts/parse_mql_calendar_bin.py`. The shipped sample file is
`assets/mql5.com-artical-22196-MetaQuotes/outputs/USD_calendar_test_res.bin`
and is generated by the article's
`assets/mql5.com-artical-22196-MetaQuotes/22196-attaches/ExportCalendarForTester-S.mq5`.

---

## 1. `MqlCalendarValue` struct (MQL5 docs)

Cited verbatim from the MQL5 documentation:

> `references/docs/01-constants/0186-constants-structures-mqlcalendar.md`
> (mirror of `https://www.mql5.com/en/docs/constants/structures/mqlcalendar`)

```
struct MqlCalendarValue
  {
    ulong                       id;                    // value ID
    ulong                       event_id;              // event ID
    datetime                    time;                  // event date and time
    datetime                    period;                // event reporting period
    int                         revision;              // revision of the
                                                       //   published indicator
                                                       //   relative to the
                                                       //   reporting period
    long                        actual_value;          // × 10^6 or LONG_MIN
    long                        prev_value;            // × 10^6 or LONG_MIN
    long                        revised_prev_value;    // × 10^6 or LONG_MIN
    long                        forecast_value;        // × 10^6 or LONG_MIN
    ENUM_CALENDAR_EVENT_IMPACT  impact_type;           // 0=NA, 1=positive,
                                                       //   2=negative
  };
```

Field-by-field (also documented in the MQL5 docs page above):

| Field | Type | Size | Notes |
|---|---|---|---|
| `id` | `ulong` | 8 | Unique per **release** of an event value. The same event on a different date has a different `id`. |
| `event_id` | `ulong` | 8 | Event family ID. First 3 digits = ISO 3166-1 numeric country code (`840` = USA, `978` = EUR, `392` = JPY). |
| `time` | `datetime` | 8 | Unix seconds (UTC epoch). Stored as the broker-server local timestamp; do NOT assume UTC. |
| `period` | `datetime` | 8 | Start of the reporting period the indicator describes (e.g. `2024-01-01 00:00:00` for a January release). |
| `revision` | `int` | 4 | 0 = first release, 1+ = revisions. |
| `actual_value` | `long` × 10⁻⁶ → `float64` | 8 | Published value × 10^6 in the binary; converted to `float` × 10⁻⁶ in the DataFrame. `LONG_MIN` (-9223372036854775808) marks "not set" and is decoded as `NaN`. |
| `prev_value` | `long` × 10⁻⁶ → `float64` | 8 | Previous period value × 10⁻⁶ or `NaN`. |
| `revised_prev_value` | `long` × 10⁻⁶ → `float64` | 8 | Revised previous period value × 10⁻⁶ or `NaN`. |
| `forecast_value` | `long` × 10⁻⁶ → `float64` | 8 | Forecast value × 10⁻⁶ or `NaN`. |
| `impact_type` | `ENUM_CALENDAR_EVENT_IMPACT` | 4 (int) | 0 = `CALENDAR_IMPACT_NA`, 1 = `CALENDAR_IMPACT_POSITIVE`, 2 = `CALENDAR_IMPACT_NEGATIVE`. Records the **actual** market reaction, NOT the pre-announced `importance` filter. |

Total struct size per MQL5 docs: **80 bytes**.

## 2. On-disk record layout

When the article's export script writes the array with
`FileWriteArray(handle, values, offset, count)`, MetaTrader serializes
each struct to **128 bytes per record** in the file. The 48-byte
difference between the documented 80-byte struct and the 128-byte
record is internal padding from the MQL5 runtime's binary write
strategy (8-byte alignment + tail padding to 128).

**Layout (little-endian, all fields unsigned except where noted):**

```
offset  size  field
   0      8   ulong    value_id           (MqlCalendarValue.id)
   8      8   ulong    event_id
  16      8   int64    time               (Unix seconds)
  24      8   int64    period             (Unix seconds)
  32      4   int32    revision
  36      8   int64    actual_value       (× 10^6, or LONG_MIN)
  44      8   int64    prev_value         (× 10^6, or LONG_MIN)
  52      8   int64    revised_prev_value (× 10^6, or LONG_MIN)
  60      8   int64    forecast_value     (× 10^6, or LONG_MIN)
  68      4   int32    impact_type        (ENUM_CALENDAR_EVENT_IMPACT)
  72     56   padding  (zero-filled)
total   128   bytes per record
```

Verification on the shipped samples:

- `USD_calendar_test_res.bin` — 1,137,664 bytes ÷ 128 = **8,888 records**
- `USD_calendar_test_res-latest.bin` — 1,176,064 bytes ÷ 128 = **9,188 records**

Both files have zero remainder → layout is exact.

A file that does not satisfy `len(file) % 128 == 0` is not in this
format and `parse_mql_calendar_bin.py` will refuse to parse it.

## 3. Write side — generating the file

The shipped exporter is `22196-attaches/ExportCalendarForTester-S.mq5`
(adapted from the article's stock script to also accept an `InpEventIDs`
filter so a single file can be limited to "NFP-related" events for the
strategy tester).

Key inputs:

```
input string                          InpCurrencies = "USD";
input datetime                        InpDateFrom   = 0;          // 0 = all
input datetime                        InpDateTo     = D'2026.08.01';
input string                          InpEventCodes = "";         // substring match
input string                          InpEventIDs   = "840010007,840020001,...";
input ENUM_CALENDAR_EVENT_IMPORTANCE  InpMinImportance = CALENDAR_IMPORTANCE_HIGH;
input string                          InpOutputFile  = "calendar_test_res.bin";
input bool                            InpUseCommonDir = true;     // → \Terminal\Common\Files\
```

The save loop is plain `FileWriteArray` on each currency's slice:

```
int file_handle = FileOpen(
    currencies[i] + "_" + filename,
    FILE_WRITE | FILE_BIN | (InpUseCommonDir ? FILE_COMMON : 0)
);
FileWriteArray(file_handle, values, offset, vls_size[i]);
FileFlush(file_handle);
FileClose(file_handle);
```

The article's stress test reports the format is "the fastest way to
access structured data — access speed is limited only by the
performance of the computer's file system." A test export of 9,009
USD high-importance events produced a 1,153,152-byte file (9,009 × 128).

## 4. Read side — consuming the file inside an EA

The reference consumer is
`22196-attaches/ImportCalendarValidation-EA.mq5`. The key line is:

```
#resource "\\Files\\USD_calendar_test_res.bin" as MqlCalendarValue USD_res_calendar_data[]
```

That single pragma is enough: MetaTrader reads the file at compile
time, maps it into the EX5 as a static resource, and rehydrates it as
a `MqlCalendarValue[]` at runtime. The EA then iterates with ordinary
array semantics — no manual struct unparsing.

When the EA is **live** (not in tester), the same code path switches to
`CalendarValueHistory()` instead of the resource. The dual path is
chosen at `OnInit()` via `MQLInfoInteger(MQL_TESTER)`.

## 5. `LONG_MIN` and the "value not set" convention

The MQL5 documentation explicitly states:

> If no value is specified, the field stores `LONG_MIN`
> (-9223372036854775808).

This applies to `actual_value`, `prev_value`, `revised_prev_value`,
and `forecast_value`. In the on-disk binary these fields are scaled
`× 10^6`, so the unset sentinel is the same `LONG_MIN` value
regardless of any actual economic magnitude.

`parse_mql_calendar_bin.py` decodes this for you:

- The four `*value` columns are exposed as **`float64`** (not `int64`)
  so the unset state can be represented.
- `LONG_MIN` → `NaN` (`float('nan')`)
- everything else → the raw value × `10^-6`

So `200000` in the binary becomes `0.2` in the DataFrame, and
`LONG_MIN` becomes `NaN`. Check for unset via `pd.isna(df['actual_value'])`
or `df['actual_value'].notna()`. **Never** divide an `int64` column
in place — the sentinel would become a meaningless float before you
get the chance to mask it.

The MQL5 source provides convenience methods that hide this:
`HasActualValue()`, `GetActualValue()` (returns `NaN` if unset), etc.
The on-disk binary does not — `parse_mql_calendar_bin.py` performs
the same conversion so Python code reads the field the same way the
MQL5 docs describe.

## 6. `impact_type` ≠ pre-announced importance

Two separate enums:

| Enum | Meaning | Where |
|---|---|---|
| `ENUM_CALENDAR_EVENT_IMPORTANCE` | Pre-announced importance (`CALENDAR_IMPORTANCE_NONE / LOW / MEDIUM / HIGH`) | `MqlCalendarEvent.importance` (the **event** descriptor, not the value) |
| `ENUM_CALENDAR_EVENT_IMPACT` | Actual market reaction after release (`NA / POSITIVE / NEGATIVE`) | `MqlCalendarValue.impact_type` (the **value** record) |

The binary file does NOT carry the pre-announced importance — that
filter was applied at export time (`InpMinImportance`). If you want
per-record importance, query `CalendarEventById(event_id, event)`
against the live MetaTrader calendar API.

## 7. `event_id` decoding (offline heuristic)

The first 3 decimal digits of `event_id` are the ISO 3166-1 numeric
country code (matching the prefix in the MQL5 docs, e.g. `840` =
United States, `978` = Eurozone, `392` = Japan, `826` = United Kingdom).

The remaining digits are an internal MQL5 counter with no public
mapping. There is **no** programmatic way to translate `event_id` to
the human-readable event name without either:

1. Calling `CalendarEventById(event_id, event)` on a live terminal, or
2. Maintaining a side table (the skill ships a small one at
   `assets/calendar-events.csv` — 6 entries that map the
   `event_id`s used by the article's export script to human names).
   The `outputs/USD_calendar_test_res*.bin` file is gitignored
   (`outputs/` is excluded by the assets `.gitignore`); regenerate it
   via `assets/mql5.com-artical-22196-MetaQuotes/22196-attaches/ExportCalendarForTester-S.mq5`.

The article's exporter pattern assumes live-terminal access at compile
time and embeds the resource; offline analysis tools (like
`parse_mql_calendar_bin.py`) work with numeric `event_id` only.

## 8. Reference parsers

`skills/mql5/scripts/parse_mql_calendar_bin.py` reads any file matching
`USD_calendar_test_res*.bin` (or an explicit path), decodes it with
`struct.unpack('<QQqqiqqqqi56x')`, and returns a `pandas.DataFrame`
with one column per MqlCalendarValue field.

The script does **NOT** shift timestamps — `time` and `period` are
written exactly as MetaTrader stores them (broker-server local Unix
seconds). To display in a specific timezone, apply
`df['time'].dt.tz_localize('Etc/GMT-3').dt.tz_convert('America/New_York')`
or similar in your own analysis code.

---

## 9. Quick checklist for any new "calendar binary" tool

1. Confirm the file size is a multiple of 128 — if not, it is not in
   this format (or the MQL5 runtime changed its alignment policy).
2. Parse the first record: `id` and `event_id` should be plausible
   values (positive ulongs ≤ 2^63), `time` should be a Unix timestamp
   in the 2007–present range.
3. Treat any value field equal to `-9_223_372_036_854_775_808` as
   "not set".
4. If you need human-readable event names, expect to ship a separate
   `event_id → title` table. There is no inline name in the binary.
5. If you need `importance` (pre-announced), query the live calendar
   API — it is not in the binary.
6. Do not interpret `time` as UTC by default — broker-server local
   timezone depends on the broker (article 22196's example broker
   appears to write server-local time into the Unix timestamp).
